Database Access in Node.js: Full Lab Guide

Lab goal

This lab turns the Week 3 lecture material into a repeatable Ubuntu workflow. You will run six small Node.js examples and compare how the same classroom dataset is accessed through direct drivers, an ORM, a query builder, and a graph traversal API.

The emphasis is not only on seeing output. The emphasis is on understanding what each setup script does, why each connection string exists, and how CRUD operations change across PostgreSQL, MongoDB and TinkerGraph.

 

Big idea

Run one folder at a time, read the terminal output, then compare the Node.js CRUD method being demonstrated. The same teaching dataset appears in different forms: rows in PostgreSQL, documents in MongoDB, and vertices plus edges in TinkerGraph.

Before you start

  • This lab is written for Ubuntu only.
  • The setup scripts are designed to be repeatable. If the database software is already installed, the script checks the existing installation and reuses it.
  • The classroom password 123456 is used only for a local teaching VM. It is not acceptable for a real deployment.
  • Most setup scripts create or refresh the teaching database, seed the dataset, copy .env.example to .env if needed, install Node.js packages, and then print useful manual test commands.

What each setup script does

Although each folder targets a different database or access tool, the scripts follow a similar pattern.

Script stage What it means
Check platform tools Install or verify Node.js and any required database runtime such as PostgreSQL, MongoDB, Java or Gremlin Server prerequisites.
Start the database service Enable and start PostgreSQL or MongoDB with systemctl, or prepare the Gremlin Server files so you can start the server manually.
Create or refresh classroom access Create the local classroom database, local user and password, or update them if they already exist.
Seed the teaching dataset Insert the same student, course and post data used by the lecture page so that the code examples and the page describe the same dataset.
Prepare Node.js files Copy .env.example to .env if the file is missing, then install npm packages.
Print manual test commands Show the exact terminal commands you can run after setup to test the database connection yourself before running the Node.js example.

How to run each folder

Open Ubuntu Terminal and move into the full lab pack. Then run each folder in order.

cd database_access_node_labs_full

PostgreSQL direct driver

cd postgresql-pg-direct
chmod +x setup_ubuntu_postgresql.sh
./setup_ubuntu_postgresql.sh
npm start
cd ..

PostgreSQL with Prisma

cd postgresql-prisma
chmod +x setup_ubuntu_postgresql_prisma.sh
./setup_ubuntu_postgresql_prisma.sh
npm start
cd ..

PostgreSQL with Drizzle

cd postgresql-drizzle
chmod +x setup_ubuntu_postgresql_drizzle.sh
./setup_ubuntu_postgresql_drizzle.sh
npm start
cd ..

MongoDB direct driver

cd mongodb-driver-direct
chmod +x setup_ubuntu_mongodb.sh
./setup_ubuntu_mongodb.sh
npm start
cd ..

MongoDB with Prisma

cd mongodb-prisma
chmod +x setup_ubuntu_mongodb_prisma.sh
./setup_ubuntu_mongodb_prisma.sh
npm start
cd ..

TinkerGraph with Gremlin

This folder has one extra requirement: you start Gremlin Server in one terminal, then run the Node.js CRUD demo in another.

cd gremlin-direct
chmod +x setup_ubuntu_tinkergraph.sh
./setup_ubuntu_tinkergraph.sh

After setup, open a second terminal and start Gremlin Server from the path printed by the script:

cd "$HOME/apache-tinkerpop-gremlin-server-3.7.4"
./bin/gremlin-server.sh conf/gremlin-server-modern.yaml

Then return to the lab folder and run the Node.js example:

cd ~/path/to/database_access_node_labs_full/gremlin-direct
npm start

How to read the terminal output

Each Node.js example tells you which access method is being used. That line is important because it explains the programming style you are about to see.

Folder Expected CRUD method label
postgresql-pg-direct SQL text executed with pool.query()
postgresql-prisma prisma.student.create/findMany/update/delete
postgresql-drizzle db.insert/select/update/delete
mongodb-driver-direct collection.insertOne/find/updateOne/deleteOne
mongodb-prisma Prisma ORM for MongoDB
gremlin-direct Direct Gremlin traversal API

Manual database checks

Before blaming the Node.js code, confirm that the database service is running and that the classroom account can connect.

PostgreSQL checks

node -v
sudo systemctl status postgresql --no-pager
PGPASSWORD=123456 psql "postgresql://university_user:123456@localhost:5432/university_db"
\dt
SELECT id, name, major FROM students ORDER BY id;
\q

MongoDB checks

sudo systemctl status mongod --no-pager
mongosh "mongodb://university_user:123456@localhost:27017/university_db?authSource=admin"
show collections
db.studentProfiles.find({}, { _id: 0, studentId: 1, name: 1, major: 1 }).pretty()
exit

Gremlin checks

cd "$HOME/apache-tinkerpop-gremlin-server-3.7.4"
./bin/gremlin.sh
:remote connect tinkerpop.server conf/remote.yaml
:remote console
g.V().count()
g.V().hasLabel('person').values('name')

How to stop programs

  • Most npm start demos run, print output, and finish by themselves.
  • If a foreground process keeps running, stop it with Ctrl + C.
  • For the Gremlin lab, stop the Gremlin Server terminal with Ctrl + C after the Node.js demo finishes.

What the lab is teaching you

  • How Ubuntu scripts prepare a local database environment safely and repeatably.
  • How the same CRUD concept appears in SQL, NoSQL and graph systems.
  • How direct drivers differ from Prisma and Drizzle in everyday Node.js code.
  • How to debug database access by checking the service, the connection string, the classroom user, and the seeded dataset in order.
