Lab goal

In Lab 1 you used direct database drivers. In this lab you will use Prisma, an ORM, to access the same PostgreSQL students table through a model. The database is still PostgreSQL. Prisma does not replace the database. It gives your Node.js code a structured model layer on top of it.

 

Why use an ORM?

An ORM maps database tables to application models. It can reduce repeated SQL strings, centralise schema knowledge, and make ordinary CRUD code easier to read. The trade-off is that you must still understand the real database underneath, especially indexes, constraints, migrations and query performance.

Step 1: Start from the lab folder

Begin from the unzipped lab root so the paths match this page.

cd ~/Downloads/lab-v2
pwd
ls

Step 2: Confirm Lab 1 PostgreSQL is working

Do not start Prisma until the database works manually. Run this check first.

sudo systemctl start postgresql
PGPASSWORD=123456 psql -v ON_ERROR_STOP=1 -h localhost -U university_user -d university_db -c "SELECT id, name, major FROM students ORDER BY id;"

If this command fails, return to Lab 1 and fix PostgreSQL first.

Step 3: Open the Prisma lab folder

cd ~/Downloads/lab-v2/lab2-postgresql-prisma
cp .env.example .env
cat .env

The connection string should point to the same database from Lab 1:

DATABASE_URL=postgresql://university_user:123456@localhost:5432/university_db?schema=public

Step 4: Install Prisma packages

The package versions are fixed in package.json: @prisma/client 6.19.3 is the runtime client and prisma 6.19.3 is the development CLI used to generate the client. The supplied package-lock.json file locks the full dependency tree, so use npm ci for repeatable installation.

npm ci

Step 5: Read the Prisma schema

Open prisma/schema.prisma. The important part is the Student model.

code prisma/schema.prisma

The line @@map("students") means the Prisma model named Student maps to the existing PostgreSQL table named students. The line @map("created_at") means the JavaScript field createdAt maps to the SQL column created_at.

Step 6: Generate Prisma Client

Prisma Client is generated code. Your application imports it and uses it to run model-based database operations.

npm run generate

Step 7: Run the Prisma CRUD demo

npm start

You should see the students table printed several times: before changes, after create, after update, and after delete. The CREATE section uses upsert, not plain create, so the demo can be run repeatedly. If S7 already exists, Prisma updates it instead of failing with a duplicate primary key error.

Step 8: Modify the Prisma code yourself

Open prisma_crud.js.

code prisma_crud.js
  • Find the upsert block. Change Henry Wilson to another fake student name.
  • Change the fake email address.
  • Run npm start again.
  • Find the filter where major contains Web. Change it to Cloud and run again.
  • Temporarily comment out the DELETE block. Run the program, then manually check PostgreSQL with SELECT id, name, major FROM students ORDER BY id;.

Step 9: Compare direct SQL and Prisma

Question Direct SQL Prisma
How does code create a student? INSERT INTO students... prisma.student.upsert() in this repeatable teaching demo
Where is the data stored? PostgreSQL table The same PostgreSQL table
What does Prisma add? No extra model layer A generated model-based client

Expected outcome

By the end of Lab 2, you should be able to explain why an ORM can make application code more organised, while still recognising that the underlying database design remains important.
